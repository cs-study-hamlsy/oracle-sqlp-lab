# 1.1.5 옵티마이저 힌트

## 학습 목표

- `FULL`, `INDEX`, `LEADING`, `USE_NL` 힌트의 역할을 구분한다.
- 힌트가 액세스 경로, 조인 순서, 조인 방식에 어떤 영향을 주는지 이해한다.
- 힌트는 실행계획으로 검증해야 하며, 무조건 성능을 좋게 만드는 도구가 아니라는 점을 이해한다.

## 교재 내용 정리

이 챕터의 핵심은 "옵티마이저는 기본적으로 스스로 판단하지만, 필요한 경우 사람이 실행 방향을 제어할 수도 있다"는 점이다.

많은 학습자가 힌트를 "빠르게 만드는 주문"처럼 오해한다. 하지만 힌트는 성능 자체를 보장하는 장치가 아니라, 옵티마이저가 고려할 경로에 방향을 주는 수단이다. 따라서 힌트의 본질은 "무조건 좋다/나쁘다"가 아니라 "어떤 상황에서 어떤 방향을 강제하는가"에 있다.

이 챕터에서 꼭 잡아야 할 메시지는 아래와 같다.

- `INDEX` 힌트는 인덱스 경로를 우선 고려하게 만든다.
- `FULL` 힌트는 풀 스캔 경로를 강하게 유도한다.
- `LEADING`은 조인 시작 순서에 영향을 준다.
- `USE_NL`은 조인 방식을 Nested Loop 쪽으로 유도한다.
- 힌트를 줬더라도 실제로 반영됐는지는 실행계획으로 확인해야 한다.

## 상세 개념 정리

### 1. 힌트란 무엇인가

힌트는 SQL 주석 형태로 SQL 안에 들어가며, 옵티마이저에게 특정 경로를 고려하거나 강제하도록 방향을 주는 문법이다.

중요한 점은 힌트가 SQL 의미를 바꾸는 것이 아니라, SQL을 수행하는 "길"에 영향을 준다는 것이다.

즉 아래 둘은 결과는 같지만, Oracle이 결과를 찾아가는 방식이 달라질 수 있다.

- 힌트 없는 SQL
- 힌트가 포함된 SQL

### 2. 단일 테이블 힌트

#### `INDEX`

`INDEX` 힌트는 특정 인덱스를 사용한 접근 경로를 우선 고려하게 만든다.

이 힌트가 잘 맞는 상황은 다음과 같다.

- 조건 선택도가 높다
- 결과 건수가 적다
- 인덱스 선두 컬럼을 조건절이 잘 활용한다

하지만 결과 집합이 너무 크면, 인덱스를 타고 다시 테이블을 반복 방문하는 비용 때문에 오히려 손해일 수 있다.

#### `FULL`

`FULL` 힌트는 테이블 전체를 읽는 경로를 유도한다.

많은 학습자가 풀 스캔을 무조건 나쁜 계획처럼 생각하는데, 이는 오해다. 결과 건수가 많거나 선택도가 낮다면 FULL SCAN이 더 단순하고 유리할 수 있다.

즉 FULL SCAN은 "나쁜 계획"이 아니라 "읽어야 할 양이 많을 때 자연스러운 계획"일 수 있다.

### 3. 조인 힌트

#### `LEADING`

`LEADING`은 조인 순서에 영향을 준다. 조인 튜닝에서 액세스 경로 못지않게 중요한 것이 "어느 집합을 먼저 읽는가"다.

작은 집합을 먼저 읽고 큰 집합을 뒤에서 붙이는 구조가 유리한 경우가 많기 때문에, `LEADING`은 조인의 출발점을 통제하는 데 의미가 있다.

#### `USE_NL`

`USE_NL`은 특정 테이블을 Nested Loop 방식으로 붙이도록 유도한다.

Nested Loop는 아래 조건에서 유리할 가능성이 높다.

- 바깥쪽 집합이 작다
- 안쪽 테이블에 조인 인덱스가 잘 준비돼 있다
- 반복 탐색 비용이 낮다

반대로 바깥쪽 집합이 너무 크면 Nested Loop는 반복 탐색이 많아져 불리할 수 있다.

## Oracle 내부 동작 원리

힌트를 이해하려면 Oracle이 원래 어떻게 판단하는지 먼저 알아야 한다.

### 1. 옵티마이저는 기본적으로 비용 기반으로 판단한다

Oracle은 통계정보를 바탕으로 아래를 추정한다.

- 읽어야 할 로우 수
- 인덱스를 탔을 때의 범위
- 조인 시 중간 결과 집합 크기
- 예상 I/O 비용

힌트는 이 판단을 완전히 없애는 것이 아니라, 특정 경로를 더 강하게 고려하게 하거나 일부 방향을 제어하는 장치다.

### 2. 왜 힌트가 항상 정답이 아닌가

힌트는 현재 데이터 분포와 통계에 맞아떨어질 때 유효하다. 하지만 시간이 지나 데이터가 커지거나 분포가 달라지면, 과거에는 좋았던 힌트가 오히려 나쁜 경로를 고정할 수 있다.

실무에서 힌트 SQL이 장기적으로 위험한 이유가 여기에 있다.

### 3. 왜 실행계획 검증이 필수인가

힌트를 썼다고 해서 반드시 그 힌트가 의도한 그대로 반영되는 것은 아니다.

- 힌트 문법이 잘못됐을 수 있다
- 별칭(alias)이 맞지 않을 수 있다
- 다른 조건 때문에 기대한 연산이 나오지 않을 수 있다

그래서 힌트 학습의 핵심은 "힌트를 썼다"가 아니라 "힌트가 실제로 어떤 계획 변화로 이어졌는가"를 검증하는 데 있다.

## 실무 관점 해석

실무에서는 힌트를 남발하면 안 된다.

### 언제 힌트를 검토하는가

- 통계정보만으로는 원하는 계획이 안정적으로 나오지 않을 때
- 특정 인덱스 사용 여부를 검증하고 싶을 때
- 조인 순서 차이가 성능에 매우 큰 영향을 줄 때
- 튜닝 실험에서 계획 차이를 의도적으로 비교할 때

### 힌트 사용 시 확인할 것

- 실제 실행계획이 힌트를 반영했는가
- 바인드 값이 달라져도 안전한가
- 통계정보를 바로잡는 쪽이 더 본질적 해결책은 아닌가
- 힌트가 SQL의 유지보수성을 해치지 않는가

## SQLP 시험 포인트

- 힌트는 옵티마이저 판단을 보조 또는 강제하는 도구입니다.
- `INDEX`가 있다고 항상 빠른 것이 아닙니다.
- 조인에서는 액세스 경로보다 "어느 집합을 먼저 읽는가"가 더 중요해질 수 있습니다.
- 힌트 문제는 "무슨 힌트가 있나"보다 "왜 그 힌트가 그 상황에서 맞는가"를 판단하는 문제가 많습니다.

## 실습 목적

- 단일 테이블에서 `FULL`과 `INDEX` 힌트 차이를 확인한다.
- 조인에서 `LEADING`, `USE_NL`이 실행계획에 미치는 영향을 비교한다.
- 힌트를 줬을 때와 안 줬을 때의 차이를 실행계획으로 검증한다.
