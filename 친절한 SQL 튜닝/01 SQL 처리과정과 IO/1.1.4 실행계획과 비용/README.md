# 1.1.4 실행계획과 비용

## 학습 목표

- 실행계획이 무엇을 보여주는지 설명할 수 있다.
- `TABLE ACCESS FULL`, `INDEX RANGE SCAN`, `TABLE ACCESS BY INDEX ROWID`의 차이를 이해한다.
- 비용(`Cost`)이 실제 실행시간 그 자체가 아니라 옵티마이저의 상대적 선택 기준이라는 점을 이해한다.
- 인덱스 컬럼 구성과 조건절이 접근 경로 선택에 어떤 영향을 미치는지 설명할 수 있다.

## 교재 내용 정리

이 챕터의 핵심은 "SQL을 작성했다"에서 끝나지 않고, Oracle이 그 SQL을 어떤 경로로 실행할지를 읽는 능력을 기르는 데 있다.

실행계획은 결과를 만드는 계산 절차의 청사진이다. Oracle은 같은 결과를 반환하는 SQL이라도 여러 가지 경로로 수행할 수 있다. 예를 들어 테이블 전체를 처음부터 끝까지 읽는 방법도 있고, 인덱스를 통해 필요한 로우 범위만 먼저 찾은 뒤 테이블을 재방문하는 방법도 있다.

이때 옵티마이저는 통계정보를 바탕으로 "어느 경로가 더 비용이 낮을 것인가"를 계산한다. 따라서 실행계획을 본다는 것은 단순히 명령어 이름을 외우는 것이 아니라, Oracle이 왜 그 길을 골랐는지 해석하는 작업이다.

이 챕터에서 특히 중요한 포인트는 아래 세 가지다.

- 실행계획은 `어떤 연산을 어떤 순서로 수행하는가`를 보여준다.
- 비용(`Cost`)은 상대 비교값이지 절대 성능 수치가 아니다.
- 인덱스가 존재한다고 해서 항상 인덱스를 타는 것이 정답은 아니다.

## 상세 개념 정리

### 1. 실행계획이란 무엇인가

실행계획은 SQL을 수행하기 위해 Oracle이 선택한 연산 트리이다.

- 어떤 오브젝트를 읽는가
- 어떤 방식으로 읽는가
- 조인이 있다면 어떤 순서와 방법으로 조인하는가
- 최종 결과를 만들기 위해 정렬, 필터링, 집계를 어떻게 하는가

즉, 실행계획을 본다는 것은 "이 SQL이 데이터를 어떤 길로 찾아가는가"를 보는 것이다.

### 2. 비용(`Cost`)의 의미

`Cost`는 초 단위 시간도 아니고, 사용자가 체감하는 응답시간과 1:1로 대응하는 값도 아니다. 옵티마이저가 여러 후보 계획을 비교하기 위해 사용하는 내부 추정값이다.

여기서 중요한 점은 다음과 같다.

- Cost가 낮으면 옵티마이저는 그 계획을 더 유리하다고 판단할 가능성이 높다.
- 하지만 Cost가 낮다고 반드시 실제 실행도 가장 빠르다는 뜻은 아니다.
- 통계정보가 부정확하면 Cost 계산도 왜곡될 수 있다.
- 따라서 Cost는 "판단 근거"이지 "절대 정답"이 아니다.

### 3. 액세스 경로의 차이

이 챕터에서 자주 보는 액세스 경로는 아래 세 가지다.

- `TABLE ACCESS FULL`
  - 테이블 전체 블록을 넓게 읽는다.
- `INDEX RANGE SCAN`
  - 인덱스에서 필요한 키 범위를 먼저 찾는다.
- `TABLE ACCESS BY INDEX ROWID`
  - 인덱스로 로우 위치를 찾은 뒤, 실제 테이블 블록을 다시 읽는다.

인덱스 경로는 "필요한 로우 수가 적고 선택도가 높을 때" 유리하다. 반대로 읽어야 할 로우가 너무 많다면 인덱스를 따라간 뒤 다시 테이블로 돌아오는 비용이 커져서 풀 스캔이 더 자연스러울 수 있다.

### 4. 복합 인덱스와 선두 컬럼

복합 인덱스는 컬럼 순서가 매우 중요하다.

예를 들어 `t_x02(deptno, job, no)` 인덱스는 아래 조건처럼 선두 컬럼부터 자연스럽게 타는 SQL에서 강점을 가진다.

- `deptno = 10`
- `job = 'MANAGER'`
- `no between 1 and 100`

반면 조건절이 인덱스 선두 컬럼과 잘 맞지 않으면, 인덱스가 있어도 효율이 떨어질 수 있다.

## Oracle 내부 동작 원리

이 챕터를 제대로 이해하려면 Oracle이 SQL을 처리하는 흐름을 함께 봐야 한다.

### 1. 파싱과 최적화

SQL이 들어오면 Oracle은 먼저 문법과 의미를 해석하고, 그 다음 옵티마이저가 가능한 실행 경로를 검토한다.

이때 옵티마이저는 아래 요소를 본다.

- 테이블 로우 수
- 컬럼 분포
- 인덱스 존재 여부
- 조건절 형태
- 예상 I/O 양

결국 실행계획은 "현재 가진 통계와 메타데이터를 바탕으로 가장 낫다고 본 선택"이다.

### 2. 왜 인덱스를 타고도 다시 느릴 수 있는가

인덱스는 바로 데이터를 담고 있는 경우도 있지만, 일반적으로는 "로우 위치"를 먼저 찾는 구조다. 그래서 인덱스를 스캔한 뒤 다시 테이블 블록을 읽는 과정이 필요하다.

이 과정은 결과 건수가 적을 때는 매우 효율적이지만, 결과가 많아질수록 랜덤 I/O가 늘어나고 비효율이 커질 수 있다. 이런 경우에는 차라리 풀 스캔이 더 단순하고 저렴할 수 있다.

### 3. 통계정보가 중요한 이유

옵티마이저는 실제 데이터를 전부 읽고 결정하지 않는다. 통계정보를 기반으로 추정한다. 따라서 통계정보가 오래됐거나 왜곡돼 있으면, 실행계획과 Cost도 현실과 어긋날 수 있다.

실무에서 "왜 이런 말도 안 되는 계획이 나왔지?"라는 상황의 상당수는 통계정보 문제와 연결된다.

## 실무 관점 해석

실무에서는 실행계획을 볼 때 단순히 연산 이름만 보면 안 된다.

### 확인해야 할 것

- Predicate Information에서 어떤 조건이 인덱스 단계에서 소화되는가
- 예상 로우 수와 실제 로우 수 차이가 큰가
- 인덱스 스캔 이후 테이블 접근량이 너무 크지 않은가
- 통계정보 수집 시점이 최신인가
- 조건절과 인덱스 선두 컬럼이 잘 맞는가

### 추가 검증 포인트

- `DBMS_XPLAN.DISPLAY` 또는 실제 실행계획 조회
- 통계정보 재수집 전후 계획 변화
- 조건 선택도 변화에 따른 경로 변화
- 동일 SQL의 바인드 값 차이에 따른 계획 변화 가능성

## SQLP 시험 포인트

- 복합 인덱스는 선두 컬럼 사용 여부가 중요합니다.
- 인덱스가 존재한다고 해서 항상 인덱스 경로가 정답은 아닙니다.
- Cost는 절대 성능값이 아니라 옵티마이저의 상대 판단값입니다.
- 실행계획 문제에서는 연산 이름만 외우는 것보다 "왜 이 경로가 선택되었는가"를 설명할 수 있어야 합니다.

## 실습 목적

- `t_x01(deptno, no)`와 `t_x02(deptno, job, no)`를 비교해 복합 인덱스 컬럼 구성이 어떤 영향을 주는지 확인한다.
- 힌트 없이 나온 계획과 힌트로 강제한 계획을 비교해, 접근 경로 차이를 눈으로 확인한다.
- FULL SCAN과 INDEX 경로의 유불리를 데이터 조건과 함께 해석한다.
