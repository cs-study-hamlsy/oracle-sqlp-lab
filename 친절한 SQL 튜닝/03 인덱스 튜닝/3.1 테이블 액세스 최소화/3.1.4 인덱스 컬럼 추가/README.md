# 3.1.4 인덱스 컬럼 추가

## 학습 목표

- 테이블 액세스 최소화를 위해 기존 인덱스에 컬럼을 추가하는 이유를 이해한다.
- 선두 컬럼 액세스 범위는 그대로여도, 인덱스 단계에서 필터링 가능한 컬럼이 늘어나면 왜 성능이 좋아지는지 파악한다.
- 실행계획에서 `INDEX RANGE SCAN -> TABLE ACCESS BY INDEX ROWID` 구조를 볼 때, 문제의 핵심이 인덱스 스캔량인지 테이블 랜덤 액세스량인지 구분할 수 있다.
- SQLP 시험과 실무에서 “인덱스를 새로 갈아엎기 어렵다면 기존 인덱스 뒤에 컬럼을 덧붙여 개선”하는 전형적인 판단 기준을 정리한다.

## 교재 내용 정리

교재는 테이블 액세스 최소화를 위한 가장 대표적인 방법으로 “인덱스에 컬럼을 추가하는 방법”을 설명한다. 핵심은 단순하다.  
기존 인덱스만으로는 원하는 로우를 찾는 범위를 크게 줄이지 못하더라도, 테이블까지 내려가기 전에 인덱스 리프 블록에서 한 번 더 걸러낼 수 있으면 불필요한 `TABLE ACCESS BY INDEX ROWID`를 크게 줄일 수 있다.

교재 예시는 `EMP_X01(DEPTNO, JOB)` 인덱스가 있을 때 아래 SQL을 처리하는 상황이다.

```sql
select *
from emp
where deptno = 30
and sal >= 2000;
```

이 SQL은 `DEPTNO = 30` 조건으로는 인덱스를 탈 수 있다. 하지만 `SAL` 컬럼이 인덱스에 없으면, `DEPTNO = 30`에 해당하는 후보 로우를 전부 테이블에서 읽은 뒤 `SAL >= 2000`인지 확인해야 한다.  
즉, 문제는 “인덱스를 못 탄다”가 아니라 “인덱스를 탄 뒤에도 테이블을 너무 많이 방문한다”는 데 있다.

이때 인덱스를 `DEPTNO + JOB + SAL`처럼 확장하면 선두 컬럼 구성은 유지하면서도 `SAL`을 인덱스 리프 블록에서 확인할 수 있다.  
중요한 점은, `JOB` 조건이 없으므로 `SAL`이 **액세스 조건**으로 승격되지는 못할 수 있다는 것이다. 그래도 `SAL`을 **인덱스 필터 조건**으로 활용해 불필요한 테이블 방문을 줄일 수 있다. 교재가 강조하는 “인덱스 스캔량은 거의 그대로인데도 성능이 좋아질 수 있다”는 말이 바로 이 의미다.

## 핵심 원리

### 1. 인덱스 튜닝의 본질은 테이블 방문량 통제다

인덱스를 탄다고 끝이 아니다. 온라인 SQL에서 성능을 무너뜨리는 주범은 종종 인덱스 스캔 자체보다도 그 뒤에 이어지는 대량의 랜덤 테이블 액세스다.  
특히 클러스터링 팩터가 좋지 않거나 결과 로우가 테이블 전역에 흩어져 있으면, 인덱스에서 로우 후보를 많이 찾는 순간 블록 I/O가 폭증한다.

따라서 DBA 관점에서는 아래 순서로 봐야 한다.

1. 인덱스 스캔으로 몇 건의 후보를 찾는가
2. 그 후보 중 실제 결과는 몇 건인가
3. 테이블을 몇 번 방문하는가
4. 그 방문이 랜덤 I/O로 얼마나 확산되는가

교재 사례는 1번보다 3번이 더 심각한 경우다.

### 2. 뒤쪽 컬럼 추가는 “범위 축소”보다 “테이블 방문 제거”에 가깝다

`(DEPTNO, JOB)` 인덱스에 `SAL`을 뒤에 붙여 `(DEPTNO, JOB, SAL)`로 만들더라도, `JOB` 조건이 빠져 있으면 `SAL`을 이용해 바로 스캔 시작점과 종료점을 정교하게 자르기 어렵다.  
그래서 실무자는 “선두가 아닌데 추가해도 의미가 있나?”라는 질문을 자주 한다.

의미가 있다. 이유는 다음과 같다.

- 기존 인덱스: `DEPTNO = 30`인 후보를 찾고, 그 후보 전부를 테이블에서 읽은 뒤 `SAL` 확인
- 컬럼 추가 후: `DEPTNO = 30`인 후보를 찾되, 인덱스에서 `SAL`을 먼저 확인하고 필요한 로우만 테이블 방문

즉, **인덱스 스캔량은 비슷할 수 있어도 테이블 랜덤 액세스 횟수는 크게 줄어든다.**

### 3. 인덱스를 새로 바꾸는 것보다 추가가 현실적인 경우가 많다

이론적으로는 `DEPTNO + SAL` 순으로 재구성하는 것이 더 좋아 보일 수 있다.  
하지만 실운영에서는 기존 `(DEPTNO, JOB)` 인덱스를 의존하는 SQL이 이미 많을 수 있다. 이때 인덱스 순서를 바꾸면 다른 SQL의 실행계획이 흔들릴 수 있다.

그래서 실무에서는 아래 판단을 자주 한다.

- 기존 인덱스 재설계는 영향 범위가 크다.
- 그러나 뒤에 컬럼 하나 추가하는 것은 상대적으로 영향 통제가 쉽다.
- 저장공간 증가와 DML 부담은 감수하되, 특정 고비용 SQL을 안정적으로 구제할 수 있다.

교재가 “가장 일반적으로 사용하는 튜닝 기법”이라고 한 이유가 여기에 있다.

## 왜 그런지

### 인덱스 스캔과 테이블 액세스를 분리해서 봐야 한다

초급 단계에서는 흔히 실행계획에 `INDEX RANGE SCAN`이 보이면 좋은 계획이라고 단정한다. 하지만 DBA는 그 다음 줄을 더 중요하게 본다.

```text
TABLE ACCESS BY INDEX ROWID
INDEX RANGE SCAN
```

이 구조에서 실제 비용을 키우는 것은 아래 중 하나다.

- 인덱스가 너무 많은 후보를 읽는다.
- 후보는 적당하지만 테이블에서 대부분 탈락한다.
- 테이블 로우가 흩어져 있어 랜덤 I/O가 심하다.

`3.1.4 인덱스 컬럼 추가`는 두 번째 문제를 다루는 장이다.  
즉, 인덱스로 찾아온 후보 대부분이 테이블에서 탈락하는 상황을 줄이는 것이다.

### 액세스 조건과 필터 조건은 다르다

SQLP 시험에서도 자주 나오는 포인트다.

- `Access Predicate`: 인덱스 탐색 범위를 결정하는 조건
- `Filter Predicate`: 읽어 온 후보를 걸러내는 조건

뒤쪽 컬럼을 추가했을 때 언제나 `Access Predicate`가 되지는 않는다.  
그렇더라도 인덱스 단계의 `Filter Predicate`가 되면 테이블 방문량을 줄이는 데 매우 큰 효과가 있다.

이 차이를 이해하지 못하면 “선두 컬럼 아니면 소용없다”는 잘못된 결론으로 가기 쉽다.

## Oracle 내부 동작 원리

### 1. 기존 인덱스만 있을 때

`IDX_ADD_COL_X1(DEPTNO, JOB)`만 있다고 가정하자.

```sql
where deptno = 30
and sal >= 2500
```

Oracle은 대략 다음 순서로 움직인다.

1. 루트/브랜치/리프를 따라 `DEPTNO = 30` 구간을 찾는다.
2. 해당 구간의 리프 엔트리를 순차적으로 읽는다.
3. 각 엔트리의 ROWID로 테이블 블록을 방문한다.
4. 테이블에서 `SAL >= 2500`을 검사한다.

즉, `SAL` 검사는 테이블에 내려가서야 가능하다.

### 2. 컬럼 추가 후

`IDX_ADD_COL_X2(DEPTNO, JOB, SAL)`로 확장하면 리프 엔트리에 `SAL`이 함께 저장된다.

1. 여전히 `DEPTNO = 30` 구간을 찾는다.
2. 리프 엔트리를 읽으면서 `SAL >= 2500`을 먼저 확인한다.
3. 조건을 통과한 엔트리만 ROWID로 테이블 방문한다.

결과적으로 테이블 방문 횟수가 “후보 건수”에서 “실제 결과 건수” 수준으로 줄어든다.

### 3. 클러스터링 팩터가 나쁠수록 효과가 크다

테이블이 `DEPTNO` 순으로 모여 있지 않고 산재해 있으면, 한 건의 후보마다 거의 한 번씩 다른 블록을 건드릴 수 있다.  
이때 불필요한 테이블 방문을 1만 번 줄이는 것은 인덱스 리프 몇 블록 더 읽는 것보다 훨씬 큰 성능 차이를 만든다.

## 실행계획 또는 튜닝 포인트 해석

이 장에서 실행계획을 볼 때는 아래를 같이 봐야 한다.

1. `INDEX RANGE SCAN`의 실제 읽은 건수
2. `TABLE ACCESS BY INDEX ROWID`의 실제 읽은 건수
3. Predicate Information에서 `SAL`이 어디에서 평가되는지

관찰 포인트는 다음과 같다.

- 기존 인덱스에서는 `SAL >= ...`가 테이블 액세스 단계에서 필터로 보일 가능성이 높다.
- 컬럼 추가 후에는 `SAL`이 인덱스 단계 predicate에 나타날 수 있다.
- 두 계획 모두 인덱스 스캔량은 비슷해도, 테이블 액세스 실제 건수는 크게 차이 날 수 있다.

즉, “인덱스 스캔을 줄였는가?”보다 “불필요한 테이블 방문을 줄였는가?”가 이 절의 성패 판단 기준이다.

## 실무 점검 포인트

- 특정 SQL이 인덱스를 잘 타는데도 느리다면 `TABLE ACCESS BY INDEX ROWID` 실제 건수를 먼저 확인한다.
- 필터 조건 컬럼이 인덱스에 없어 테이블에서 대량 탈락하는지 본다.
- 기존 인덱스에 뒤쪽 컬럼 추가 시, 저장공간 증가와 DML 부하 증가를 함께 평가한다.
- 기존 인덱스를 참조하는 다른 SQL이 많은지 확인한다. 순서 변경보다 컬럼 추가가 안전한 경우가 많다.
- 컬럼 추가 후 통계정보 재수집과 실행계획 안정성을 반드시 검증한다.
- 클러스터링 팩터가 매우 나쁜 인덱스라면, 같은 수의 테이블 방문이라도 I/O 영향이 더 커질 수 있음을 감안한다.

## SQLP 시험 포인트

- 인덱스 뒤쪽 컬럼 추가의 핵심 목적은 테이블 랜덤 액세스 최소화다.
- 선두 컬럼이 아니어도 인덱스 필터 조건으로 활용되면 충분한 성능 개선 효과가 있다.
- 액세스 조건과 필터 조건을 구분해야 한다.
- 인덱스 스캔량은 그대로여도 성능이 좋아질 수 있다.
- 인덱스 재구성보다 컬럼 추가가 운영 영향이 적은 현실적 대안일 수 있다.

## 관찰 결과 해석 포인트

- 기존 인덱스에서는 `DEPTNO = 30` 후보 전부가 테이블로 내려가는지 본다.
- `SAL` 컬럼 추가 후 인덱스 단계에서 먼저 걸러져 테이블 방문량이 결과 건수에 가까워지는지 본다.
- 인덱스 스캔량은 비슷해도 elapsed time과 logical read가 줄어드는지 본다.
- 이 개선이 “인덱스를 더 잘 탄 것”이 아니라 “테이블을 덜 간 것” 때문임을 분리해서 해석한다.

## 한 줄 정리

`3.1.4 인덱스 컬럼 추가`의 본질은 “선두 컬럼 범위를 더 잘 자르기”가 아니라, **이미 찾은 후보 중 불필요한 테이블 방문을 인덱스 단계에서 제거하는 것**이다.
