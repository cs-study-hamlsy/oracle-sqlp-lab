# 3.1.5 인덱스만 읽고 처리

## 학습 목표

- 테이블 액세스를 줄이는 수준을 넘어, 아예 인덱스만 읽고 SQL을 끝내는 구조를 이해한다.
- 커버드 인덱스(`Covered Index`)와 커버드 쿼리(`Covered Query`)의 의미를 Oracle 관점에서 설명할 수 있다.
- 필터 조건으로 버려지는 로우가 거의 없을 때는 왜 단순 “컬럼 추가”만으로 충분하지 않고, select/group by/order by에 필요한 컬럼까지 인덱스에 포함해야 하는지 파악한다.
- Oracle에는 SQL Server의 `INCLUDE` 인덱스가 없으므로, 복합 인덱스 설계로 같은 효과를 어떻게 일부 구현하는지 이해한다.

## 교재 내용 정리

교재는 앞 절에서 “필터 조건 컬럼을 인덱스에 추가해 불필요한 테이블 액세스를 줄이는 방법”을 설명한 뒤, 이번 절에서는 한 단계 더 나아가 **인덱스만 읽고 SQL을 처리하는 방법**을 제시한다.

핵심 문제의식은 이렇다.

- 인덱스를 통해 후보를 찾는 과정은 괜찮다.
- 테이블에 내려간 뒤 필터로 버려지는 로우도 별로 없다.
- 그런데 결과 로우 수 자체가 많아서, 결국 테이블 랜덤 액세스가 많이 발생한다.

이때는 필터 컬럼을 하나 더 추가하는 정도로는 큰 개선이 어렵다.  
애초에 테이블을 방문하지 않도록, 쿼리에 필요한 컬럼을 전부 인덱스에 담아 **커버드 쿼리**로 바꾸는 것이 더 강력한 해법이다.

교재 예시의 취지는 다음과 같다.

```sql
select 부서번호, sum(수량)
from 판매집계
where 부서번호 like '123%'
group by 부서번호;
```

기존에 `부서번호` 단일 인덱스만 있다면, `부서번호 like '123%'` 조건으로 인덱스를 타더라도 `sum(수량)`을 계산하려면 각 로우마다 테이블에 내려가 `수량`을 읽어야 한다.  
하지만 인덱스를 `(부서번호, 수량)`으로 만들면 필요한 정보가 인덱스에 모두 있으므로 테이블 방문이 사라질 수 있다.

## 핵심 원리

### 1. 테이블 액세스를 줄이는 것과 없애는 것은 차원이 다르다

`3.1.4`에서는 불필요한 테이블 방문을 줄이는 것이 목표였다.  
`3.1.5`에서는 필요한 결과를 내는 데 **테이블이 전혀 필요 없도록 만드는 것**이 목표다.

즉, 차이는 아래와 같다.

- 컬럼 추가: 테이블 방문은 남지만 횟수를 줄인다.
- 인덱스만 읽고 처리: 테이블 방문 자체를 없앤다.

온라인 SQL에서 랜덤 I/O는 가장 비싼 작업 중 하나다.  
따라서 테이블 액세스를 완전히 제거할 수 있다면 성능 개선 폭은 훨씬 커진다.

### 2. 커버드 인덱스의 조건

인덱스만 읽고 처리하려면 쿼리에 필요한 정보가 모두 인덱스에 있어야 한다. 보통 아래 항목을 모두 점검한다.

- `WHERE` 절에서 필요한 컬럼
- `SELECT` 절에서 반환하거나 집계에 사용하는 컬럼
- `GROUP BY` 또는 `ORDER BY`에 필요한 컬럼

이 중 하나라도 빠지면 Oracle은 결국 테이블을 방문해 값을 보충해야 한다.

### 3. “필터 손실이 적은 SQL”일수록 커버드 전략이 중요하다

앞 절의 인덱스 컬럼 추가는 “테이블에서 많이 탈락하는 SQL”에 강하다.  
반대로 이번 절은 “테이블에 내려가면 거의 다 살아남는 SQL”에 강하다.

예를 들어 `부서번호 like '123%'`로 5만 건을 찾았고, 그 5만 건이 대부분 집계에 사용된다면 아래와 같다.

- 필터 컬럼 추가만 한 경우: 5만 번 가까운 테이블 액세스 발생
- 인덱스만 읽고 처리하는 경우: 5만 건을 인덱스에서만 읽고 끝냄

즉, 결과 건수가 많을수록 커버드 인덱스의 위력이 커진다.

## 왜 그런지

### 수량 집계는 인덱스에 수량이 없으면 반드시 테이블을 간다

```sql
select dept_code, sum(qty)
from t
where dept_code like '123%'
group by dept_code;
```

위 SQL에서 `dept_code` 인덱스만 있으면, Oracle은 인덱스에서 조건에 맞는 ROWID를 얻을 수 있다.  
하지만 `sum(qty)`를 계산하려면 `qty` 값을 읽어야 하므로 결국 테이블을 방문한다.

문제는 이 방문이 “필요 없는 로우를 걸러내기 위해서”가 아니라 “필요한 값을 가져오기 위해서” 발생한다는 점이다.  
이 경우에는 인덱스 뒤에 `qty`를 붙여 커버드 상태를 만드는 것이 가장 직접적인 해결책이다.

### 커버드 인덱스는 논리적으로 “작은 요약 테이블”과 비슷하다

인덱스는 기본적으로 키 값과 ROWID를 저장하는 구조지만, 복합 인덱스에 필요한 컬럼을 더 넣어두면 특정 SQL 관점에서는 작은 조회 전용 자료구조처럼 동작한다.

DBA가 실무에서 얻는 장점은 명확하다.

- 테이블 랜덤 액세스 제거
- 버퍼 캐시 부담 감소
- 클러스터링 팩터 영향 감소
- 응답시간 안정화

다만 대가도 있다.

- 인덱스 크기 증가
- DML 시 추가 유지비용
- 너무 많은 컬럼을 넣으면 오히려 범용성이 떨어짐

즉, “무조건 많이 넣자”가 아니라 “자주 느린 핵심 SQL을 커버할 만큼만 넣자”가 실무 원칙이다.

## Oracle 내부 동작 원리

### 1. 단일 컬럼 인덱스일 때

`IDX_INDEX_ONLY_X1(DEPT_CODE)`만 있으면 Oracle은 다음처럼 움직인다.

1. `DEPT_CODE LIKE '123%'` 범위를 인덱스에서 찾는다.
2. 각 엔트리의 ROWID를 따라 테이블을 방문한다.
3. 테이블에서 `QTY`를 읽는다.
4. `GROUP BY DEPT_CODE`, `SUM(QTY)`를 수행한다.

즉, 인덱스는 접근 경로일 뿐이고 데이터는 여전히 테이블에서 가져온다.

### 2. 복합 인덱스로 커버했을 때

`IDX_INDEX_ONLY_X2(DEPT_CODE, QTY)`가 있으면 다음이 가능해진다.

1. `DEPT_CODE LIKE '123%'` 범위를 인덱스에서 찾는다.
2. 같은 인덱스 엔트리에서 `QTY` 값을 바로 읽는다.
3. 테이블을 방문하지 않고 집계를 수행한다.

이 경우 실행계획에서 `TABLE ACCESS BY INDEX ROWID`가 사라질 수 있다.  
이 한 줄이 사라지는 것이 바로 이 절의 핵심 성과다.

### 3. ORDER BY까지 인덱스가 해결할 수 있는가

교재는 SQL Server의 `INCLUDE` 인덱스를 소개하며, Oracle과의 차이를 설명한다.  
Oracle 복합 인덱스는 뒤쪽 컬럼도 브랜치/리프 탐색 및 정렬 순서에 영향을 줄 수 있지만, SQL Server `INCLUDE` 컬럼은 리프 레벨 저장 중심이라 목적이 다르다.

실무적으로 기억할 포인트는 다음이다.

- Oracle: 복합 인덱스 컬럼은 인덱스 구조 자체에 포함된다.
- SQL Server `INCLUDE`: 탐색 키가 아닌 리프 저장 컬럼을 따로 둔다.
- Oracle에는 같은 문법이 없으므로, 필요한 경우 복합 인덱스로 커버 효과를 만든다.

## 실행계획 또는 튜닝 포인트 해석

이번 절에서 좋은 계획의 특징은 분명하다.

- `INDEX RANGE SCAN` 또는 상황에 따라 `INDEX FULL SCAN`
- `TABLE ACCESS BY INDEX ROWID`가 없음
- 집계나 정렬이 있더라도 인덱스 정보만으로 처리 가능

나쁜 계획은 보통 아래 형태다.

```text
HASH GROUP BY or SORT GROUP BY
TABLE ACCESS BY INDEX ROWID
INDEX RANGE SCAN
```

여기서 꼭 따져야 할 질문은 하나다.

“테이블을 방문한 이유가 필터 때문인가, 아니면 필요한 컬럼이 인덱스에 없어서인가?”

이번 절의 문제는 후자다.

## 실무 점검 포인트

- 성능 문제 SQL가 많은 결과 건수를 읽는 집계/조회인지 확인한다.
- 필터로 거의 버려지지 않는다면, 단순 필터 컬럼 추가보다 커버드 인덱스가 더 유효한지 검토한다.
- select/group by/order by 컬럼을 모두 점검해 테이블 방문 원인을 분해한다.
- 인덱스 폭 증가에 따른 DML 부담, 리프 블록 증가, 캐시 효율 저하를 같이 평가한다.
- 너무 많은 컬럼을 넣어 “한 SQL만을 위한 비대한 인덱스”가 되지 않도록 주의한다.
- 배치성 대량 집계라면 애초에 전체 스캔이 더 유리한지 함께 판단한다.

## SQLP 시험 포인트

- 커버드 쿼리는 테이블 액세스 없이 인덱스만 읽고 처리하는 쿼리다.
- 커버드 인덱스는 해당 쿼리에 필요한 컬럼을 모두 포함하는 인덱스다.
- 필터 손실이 적고 결과 건수가 많을수록 커버드 전략의 효과가 크다.
- Oracle에는 SQL Server의 `INCLUDE` 인덱스 문법이 없다.
- 복합 인덱스는 탐색, 정렬, 커버링을 동시에 고려해야 한다.

## 관찰 결과 해석 포인트

- 단일 컬럼 인덱스에서는 `QTY`를 읽기 위해 테이블을 방문하는지 본다.

## 한 줄 정리

`3.1.5 인덱스만 읽고 처리`의 핵심은 **필터링을 잘하는 인덱스**가 아니라, **쿼리에 필요한 데이터를 스스로 다 들고 있는 인덱스**를 만드는 데 있다.
