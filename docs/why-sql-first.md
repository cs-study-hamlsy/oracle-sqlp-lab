# 왜 SQL 먼저인가

## 결론

SQLP, 실행계획, 인덱스, 통계, 조인, 튜닝 학습이라면 C나 Pro*C보다 SQL을 직접 실행하는 방식이 훨씬 효율적입니다.

## 이유

### 1. 학습 목표와 도구가 일치한다

이 저장소의 핵심 목표는 애플리케이션 개발이 아니라 아래 항목입니다.

- SQL 처리 과정 이해
- 실행계획 해석
- 인덱스 설계와 선택도 판단
- 통계정보와 옵티마이저 판단 이해
- 조인 방식 비교
- SQL 튜닝 사고방식 훈련

이 목표는 SQL만으로 대부분 재현 가능합니다.

### 2. 불필요한 인지 부하가 줄어든다

C/Pro*C를 끼우면 아래 요소가 추가됩니다.

- 컴파일러
- Oracle Client
- 헤더/라이브러리 경로
- 전처리
- 빌드 스크립트

이것은 DBA/SQLP 학습의 본질이 아닙니다.

### 3. 실험 반복 속도가 빠르다

SQL Developer에서 바로 수정하고 다시 실행하는 편이
Pro*C 수정 -> 전처리 -> 컴파일 -> 실행보다 훨씬 빠릅니다.

튜닝 학습은 반복 관찰이 핵심이라 반복 비용이 낮아야 합니다.

### 4. 교재 맥락을 유지한 채 실습할 수 있다

SQL-only 구조는 교재 안의 파트/주제 폴더에 그대로 둘 수 있어,
출처와 실습을 분리하지 않아도 됩니다.

## 예외적으로 C/Pro*C가 유효한 경우

- OCI 연결 자체를 배우고 싶을 때
- 배열 fetch, bulk 처리, host variable, embedded SQL을 배우고 싶을 때
- 애플리케이션 계층에서 SQL 호출 비용을 같이 보고 싶을 때

즉, C/Pro*C는 "DB 애플리케이션 연동"이 목표일 때 의미가 있습니다.

## 이 저장소의 기본 정책

- 기본 실습은 SQL-only
- C/Pro*C는 레거시 또는 보조 실습
- 새 실습은 해당 교재의 파트/주제 폴더 안에 SQL 중심 구조로 작성
