# 프로젝트 구조 진단 및 개편안

## 1. 현재 구조의 핵심 문제

### 문제 1. 실습 진입점이 SQL이 아니라 실행 환경이다

현재 저장소는 학습자가 가장 먼저 봐야 할 SQL과 실행계획보다 아래 항목이 먼저 눈에 들어옵니다.

- `run.ps1`
- `proc`
- C 컴파일러
- Oracle Client
- 환경변수 로더

이 구조는 DBA, SQL 튜닝, SQLP 학습용 저장소의 우선순위와 맞지 않습니다.

SQLP 학습자는 보통 아래 질문으로 시작합니다.

- 이 SQL은 무엇을 검증하는가?
- 왜 이 인덱스를 만들었는가?
- 실행계획에서 무엇을 봐야 하는가?
- FULL SCAN과 INDEX RANGE SCAN 중 무엇이 더 합리적인가?

그런데 현재 구조는 아래 질문을 먼저 유도합니다.

- 어떻게 컴파일하지?
- Pro*C가 깔려 있어야 하나?
- `run.ps1`을 어디서 실행하지?

이것은 학습 효율을 크게 떨어뜨립니다.

### 문제 2. 교재명 중심 구조는 학습 재사용성에 약하다

`친절한 SQL 튜닝`, `오라클 성능 고도화 원리와 해법 1`, `오라클 성능 고도화 원리와 해법 2` 구조는 출처 관리에는 좋습니다.

하지만 실제 복습과 누적 학습은 보통 아래 기준으로 이뤄집니다.

- SQL 처리 과정과 I/O
- 실행계획
- 인덱스
- 조인 메커니즘
- 통계정보
- 힌트
- Lock / Transaction

즉, 책 이름보다 주제 이름 기준 탐색이 훨씬 빠릅니다.

### 문제 3. 실습 설명이 부족하다

현재 일부 README는 내용이 있지만, SQL 파일 자체만 열었을 때 아래 정보가 바로 보이지 않습니다.

- 왜 이 데이터를 만들었는가
- 어떤 조건 선택도를 의도했는가
- 어떤 실행계획 변화를 보려는가
- 결과를 어떻게 해석해야 하는가

학습용 저장소라면 SQL 스크립트도 문서여야 합니다.

### 문제 4. 실행 결과 해석 프레임이 약하다

성능 실습은 보통 아래 순서로 정리돼야 합니다.

1. 실행계획
2. 예상 원인
3. 검증 방법
4. 개선 방향

현재 저장소는 실습은 있으나, 이 프레임이 구조 차원에서 강제되지 않습니다.

### 문제 5. SQLP와 실무 DBA 관점이 구조에 충분히 드러나지 않는다

좋은 SQLP 실습 저장소는 단순 예제 모음이 아니라 아래를 드러내야 합니다.

- 옵티마이저가 왜 그 경로를 택했는지
- 통계정보가 어떤 판단 근거를 제공하는지
- 인덱스 선두 컬럼과 조건절 형태가 왜 중요한지
- 실무에서는 어떤 뷰와 플랜을 추가 확인해야 하는지

현재 구조는 아직 "실습 보관소"에 가깝고, "학습 체계"로는 덜 다듬어져 있습니다.

## 2. 개편 원칙

- 기본 진입점은 SQL Developer
- SQL 파일만 열어도 실습 목적이 이해되어야 함
- 한 실습은 한 질문에 집중
- README는 개념과 해석 중심
- 교재 출처와 실제 실습 위치를 분리하지 말고, `교재 > 파트 > 주제 > sql` 구조 안에서 학습 경로를 통일

## 3. 새 구조가 학습에 유리한 이유

### SQL만으로 바로 실행 가능

컴파일 없이 SQL Developer에서 순서대로 실행할 수 있어 진입장벽이 낮습니다.

### 학습 흐름이 선명함

`01_setup.sql -> 02_baseline -> 03_comparison -> 04_practice` 구조는 복습이 쉽고 반복 학습에도 강합니다.

### SQL 파일이 곧 실습 가이드

스크립트 상단 주석에 실습 목적과 체크 포인트를 넣으면 README를 안 열어도 핵심을 놓치지 않습니다.

### 튜닝 사고방식이 자연스럽게 내재화됨

실행계획 비교, 인덱스 선택성, 힌트 강제, 조인 순서 해석을 반복하면서 SQLP식 사고가 자리 잡습니다.

## 4. 앞으로 권장하는 표준

각 실습 폴더는 최소 아래 구성으로 관리하는 것을 권장합니다.

```text
주제 폴더/
├─ README.md
├─ 01_setup.sql
├─ 02_baseline.sql
├─ 03_comparison.sql
├─ 04_practice.sql
└─ 99_cleanup.sql
```

README에는 아래 항목을 유지하는 것이 좋습니다.

- 실습 목적
- 핵심 개념
- 실행 순서
- 기대 실행계획
- 관찰 결과
- 해석 포인트
- 실무 추가 확인 사항
- SQLP 시험 판단 포인트

## 5. 냉정한 총평

현재 저장소는 방향은 좋지만, 학습자 경험 관점에서 아직 덜 정리되어 있습니다.

- 장점: Oracle 실습 의도는 분명하고, 힌트/실행계획 같은 좋은 주제를 이미 잡고 있습니다.
- 약점: 실습 콘텐츠보다 실행 도구 체인이 전면에 나와 있습니다.
- 가장 큰 개선 포인트: "개발용 구조"에서 "학습용 구조"로 중심축을 옮겨야 합니다.

이번 개편은 그 첫 단계입니다.
